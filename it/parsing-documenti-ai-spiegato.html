Parsing di Documenti con AI: Da PDF a Dati Strutturati

Il document parsing è il processo che trasforma documenti non strutturati (PDF, scansioni, immagini) in dati strutturati (JSON, CSV, database). Ecco come funziona con l'AI moderna e perché è diverso dall'OCR tradizionale.

Definizione: cos'è il document parsing?

Il document parsing (o data extraction from documents) è il processo automatizzato di lettura di un documento in formato non strutturato e conversione delle informazioni in un formato strutturato e machine-readable.

Definizione formale

Il document parsing è il processo di estrazione automatizzata di entità strutturate da documenti non strutturati o semi-strutturati, utilizzando tecnologie di computer vision, riconoscimento ottico dei caratteri e/o modelli linguistici, al fine di produrre dati machine-readable che possono essere integrati in sistemi informativi aziendali.

Come funziona il parsing con Vision Language Model

I VLM moderni (Qwen 2.5-VL, LLaMA 3.2-Vision, Mistral Pixtral) rivoluzionano il document parsing perché non "leggono" il documento carattere per carattere — lo vedono come lo vede un umano.

Il pipeline tecnico:

  1. Conversione: PDF → immagini PNG (200-300 DPI per documenti standard). Una pagina = un'immagine.
  2. Encoding visivo: L'immagine viene divisa in patch (pezzi) di 14×14 pixel e codificata come sequenza di token visivi comprensibili al modello.
  3. Prompt strutturato: Il modello riceve immagine + istruzioni: "Estrai i seguenti campi in formato JSON: fornitore, P.IVA, numero fattura, data, importo..."
  4. Comprensione contestuale: Il modello integra la comprensione visiva (posizione dei campi, struttura della tabella) con la conoscenza linguistica (semantica dei campi finanziari italiani).
  5. Output JSON: Il modello produce un oggetto JSON strutturato con tutti i campi richiesti.

Sfide specifiche dei documenti italiani

Formato degli importi

In Italia il separatore decimale è la virgola (€1.234,56). Molti sistemi stranieri usano il punto. Il parser deve gestire entrambi i formati e non confondere mai il separatore delle migliaia con quello decimale. Un errore qui produce importi 1000x sbagliati.

Aliquote IVA multiple

Una singola fattura italiana può avere righe con aliquote diverse: 22% (standard), 10% (alimentari, ristrutturazioni), 4% (beni di prima necessità), 0% (esenti). Il parser deve estrarre l'imponibile e l'IVA per ogni aliquota separatamente.

FatturaPA XML vs PDF

Le fatture passive digitali arrivano dall'SDI in formato XML strutturato. Per queste, non serve il VLM: si fa parsing diretto dell'XML. Solo per le fatture in PDF (ancora molte, specialmente da fornitori esteri o piccoli artigiani) serve il VLM.

DDT con tabelle complesse

I DDT possono avere tabelle con 50+ righe, colonne variabili (a volte peso netto, a volte no), e unità di misura non standardizzate. L'estrazione corretta delle righe è la sfida più difficile per qualsiasi sistema di parsing.

JSON Schema enforcement: output sempre strutturato

Uno dei problemi dei VLM è che a volte producono output non valido. Per garantire affidabilità in produzione:

  • Schema validation: Ogni output JSON viene validato contro uno schema predefinito. Campi mancanti o tipi errati → retry con prompt rafforzato.
  • Valori null espliciti: Se un campo non è presente, il modello restituisce null — non omette il campo, non inventa valori.
  • Type coercion: Importi sempre come float, date sempre ISO 8601, booleani sempre true/false. Nessuna stringa ambigua.
  • Hallucination detection: Confronto cross-field: se l'IVA estratta non corrisponde al 22% dell'imponibile, è probabile un errore. Il documento va in revisione.
